Financial Compliance Agent

A production-grade AI agent that autonomously analyses financial transactions, detects anomalies, retrieves compliance policies, and produces structured reports — all running in Docker with a single command.

?What This Project Does

You submit a plain-English task like "Review recent transactions for account ACC-0042 and flag any compliance issues." The system handles everything from there — no human in the loop.

🔍

Fetches Real Data

Queries PostgreSQL for the account's recent transactions, balance history, and customer profile.

⚡

Detects Anomalies in Python

A deterministic rules engine runs before the LLM — catching structuring, velocity spikes, geographic risk, and large transactions with real database UUIDs.

📚

Retrieves Compliance Policies

Searches a vector store of compliance documents (AML thresholds, velocity checks, geographic risk rules) using semantic similarity — this is RAG.

🤖

LLM Writes the Narrative

The LLM only writes the explanation. It cannot invent transaction IDs because the rules engine already found the real ones and injected them into context.

🚨

Auto-Escalates Critical Cases

Critical or high-severity anomalies are automatically written to an escalations table for the compliance team to review.

📋

Full Audit Trail

Every tool call, LLM decision, and guardrail check is recorded in an audit log with timestamps and step numbers.

💡 The dashboard and /metrics are the monitoring layer. The real output is the structured compliance report stored in the tasks table, accessible via GET /tasks/{id}.

1System Architecture

Four Docker containers work together. The API accepts requests immediately and delegates heavy work to the worker via Redis Streams.

2Request Flow — Step by Step

What happens from the moment you POST a task to when the result is ready.

1
POST /tasks → FastAPI You send {"account_id": "ACC-0001", "description": "Review recent transactions..."}. The API creates a task row in Postgres (status: pending) and publishes a message to a Redis Stream. It returns the task_id immediately — it does not wait for the agent to finish.
2
Worker consumes the stream message The background worker process is always listening to the Redis Stream. It picks up the message, updates the task status to running, and starts the agent loop.
3
Pre-Analyzer runs first (Python rules engine) Before touching the LLM, the agent calls get_account_transactions and passes the results through the Python pre-analyzer. It runs 4 deterministic rules and produces findings with real database UUIDs — structuring, velocity, geographic risk, large amounts.
4
Findings are injected into the LLM context The pre-analyzer findings are formatted as a structured summary and injected into the conversation as a user message before the LLM is called. The LLM cannot invent IDs because the real ones are already in the context.
5
RAG retrieves relevant compliance policies The agent calls search_compliance_policies. It generates a vector embedding of the query and runs a cosine similarity search against the policies stored in pgvector, returning the top matching policy chunks.
6
LLM writes the narrative With real anomaly findings, real transaction IDs, and real policy excerpts all in context, the LLM writes the compliance narrative and produces a structured JSON final answer. Temperature is 0 for determinism.
7
Output is validated and saved The output validator confirms the JSON structure is correct (Pydantic). Critical anomalies are auto-escalated to the escalations table. The full result is saved to the task row. Every step is recorded in audit_log.
8
Poll GET /tasks/{id} You poll the endpoint until status is completed. The response contains the full structured report including anomaly list, severity ratings, policy references, and narrative summary.

3Key Components

Agent Loop (src/agent/loop.py)

The core reasoning engine. Runs up to 8 tool-call steps. Each step is logged to audit_log with timing. It injects real transaction IDs as an anchor message so the LLM cannot hallucinate values.

Pre-Analyzer (src/agent/pre_analyzer.py)

Deterministic Python rules that run before the LLM. Returns structured findings with guaranteed real database UUIDs. This is the primary anti-hallucination mechanism.

🏗 Structuring Detection

Flags transactions between $8,000–$9,999 — classic "smurfing" to stay just under the $10K reporting threshold.

⚡ Velocity Check

More than 5 transactions within any 24-hour sliding window on the same account.

🌍 Geographic Risk

Transfers to high-risk country codes: NG, IR, KP, SY, CU, MM, BY.

💰 Large Amount

Single transaction exceeding 3× the account's median transaction amount.

RAG Pipeline (src/rag/)

Compliance policy markdown files are chunked, embedded via Ollama's all-minilm model, and stored as vectors in PostgreSQL with the pgvector extension. At query time, the agent's question is embedded and the closest policy chunks are returned via cosine similarity search.

Tool Registry (src/tools/registry.py)

All tools the LLM can call are defined here with JSON Schema descriptions. The agent loop dispatches tool calls to real async Python functions that query the database.

// Tools available to the LLM agent:
"get_account_transactions"   → fetch last N transactions for an account
"get_account_details"        → account balance, risk tier, customer name
"search_compliance_policies" → RAG search against pgvector policy store
"get_flagged_transactions"   → transactions already flagged as anomalies

Events System (src/events/)

The API and worker communicate via Redis Streams — a persistent, ordered message log. The producer appends task messages; the consumer reads them with a consumer group so no message is ever lost even if the worker crashes.

LLM Adapter (src/llm/adapter.py)

A unified interface supporting both Ollama (local, free) and OpenAI (cloud, faster). Switch between them by changing LLM_PROVIDER in .env.docker — no code changes needed.

4Guardrails & Safety

Five layers of protection wrap every agent execution, enforced in src/guardrails/.

🔒 Tool Allowlist

allowlist.py

Enforces read-before-write ordering. The agent cannot call flag_anomaly without first calling get_account_transactions. Prevents blind writes.

⏱ Rate Limiter

rate_limiter.py

Caps write operations per task (max 5 anomaly flags, 3 policy searches). Prevents runaway agents from flooding the database.

🕵️ PII Scrubber

pii_scrubber.py

Strips SSNs, credit card numbers, phone numbers, and email addresses from data before it is sent to the LLM. Regex + length-gated for performance.

✅ Output Validator

output_validator.py

Validates the LLM's final JSON answer against a Pydantic schema before saving. If the schema doesn't match, the agent sends a correction request.

🚨 Escalation Handler

escalation.py

Critical and high-severity anomalies are automatically written to the escalations table for the compliance team. Viewable at GET /tasks/escalations/open.

Anti-Hallucination Architecture

The system uses a hybrid approach to prevent the LLM from inventing data:

1. Python rules engine detects anomalies → real UUIDs guaranteed
2. Findings injected into LLM context   → LLM only writes narrative
3. ID anchor message sent after fetch    → lists every valid transaction ID
4. Temperature = 0                        → no creative freedom
5. Output validator checks IDs           → rejects any invented values

5API Reference

Method Endpoint Description
POST /tasks Submit a new analysis task. Returns task_id immediately.
GET /tasks/{id} Poll for task status and result. Status: pending → running → completed.
GET /tasks/{id}/audit Full audit trail — every tool call and LLM step with timing.
GET /tasks/escalations/open All open (unresolved) critical escalations, sorted by severity.
GET /health Health check — confirms database connectivity and environment.
GET /metrics Prometheus metrics (task counts, anomaly breakdown, LLM calls).
GET /dashboard HTML dashboard showing recent tasks and open escalations. Auto-refreshes every 30s.

6Quick Start

Prerequisites

Docker Desktop
Ollama (or OpenAI key)
Make
Git

1 — Clone & Configure

git clone https://github.com/your-org/financial-agent-system
cd financial-agent-system

# .env.docker is pre-configured for Ollama
# To use OpenAI instead, edit .env.docker:
#   LLM_PROVIDER=openai
#   OPENAI_API_KEY=sk-...

2 — Pull the embedding model (Ollama only)

ollama pull all-minilm   # for policy embeddings
ollama pull llama3.2:3b  # for the agent LLM

3 — Build & Launch

make build   # builds Docker images
make up      # starts postgres, redis, api, worker
make seed    # seeds 50 accounts, 5000 transactions, policy embeddings

4 — Submit Your First Task

# Submit
curl -s -X POST http://localhost:8000/tasks \
  -H "Content-Type: application/json" \
  -d '{"account_id":"ACC-0001","description":"Review transactions and flag anomalies"}'

# Poll (replace with your task_id)
curl -s http://localhost:8000/tasks/<task-id> | python3 -m json.tool

✅ Open http://localhost:8000/dashboard to watch tasks process in real time.

7Technology Stack

🐍 Python 3.14

FastAPI for the async HTTP API. asyncpg for non-blocking Postgres queries. Pydantic v2 for schema validation everywhere.

🐘 PostgreSQL 17 + pgvector

Primary datastore for tasks, transactions, audit logs, escalations. The pgvector extension adds cosine similarity search for RAG policy retrieval.

⚡ Redis Streams

Durable, ordered message queue between the API and worker. Consumer groups ensure exactly-once delivery even if the worker restarts.

🦙 Ollama / OpenAI

Ollama runs LLMs locally (llama3.2:3b) — free, private. Switch to OpenAI gpt-4o-mini for production quality with a single env var change.

🐳 Docker Compose

Four-container setup with health checks, depends_on conditions, non-root security, and volume persistence. One command to launch everything.

📊 Prometheus Metrics

DB-backed metrics endpoint (not in-process counters) so the API and worker share the same numbers. Plug into Grafana for dashboards.

🧪 pytest

71 tests across unit and integration suites. Covers pre-analyzer rules, all 5 guardrails, API endpoints, and edge cases.

📝 Structured Logging

JSON log output with bound context (task_id, step, tool). Ready to ship to any log aggregation system (Datadog, Loki, CloudWatch).

8How It Was Built — Phase by Phase

The project was built incrementally across 6 phases, each adding a complete layer of production functionality.

Phase 1 — Foundation

Project scaffold, Postgres schema (tasks, transactions, anomalies, audit_log, escalations, policies), async DB connection pool, Pydantic models, FastAPI app skeleton.

Phase 2 — The Agent

LLM adapter (Ollama + OpenAI), tool registry with 4 real SQL-backed tools, core agent reasoning loop, RAG pipeline with pgvector, 5,000 seeded transactions across 50 accounts.

Phase 3 — Guardrails & Anti-Hallucination

Python pre-analyzer (4 deterministic rules), tool allowlist, rate limiter, PII scrubber, Pydantic output validator, auto-escalation handler. All 5 layers integrated into the agent loop.

Phase 4 — Async Event Streaming

Redis Streams producer/consumer, background worker process, API decoupled to return immediately, consumer groups for fault tolerance.

Phase 5 — Observability

Structured JSON logging throughout, DB-backed Prometheus metrics (cross-process safe), server-rendered HTML dashboard with auto-refresh.

Phase 6 — Testing & Docker

71 pytest tests (unit + integration), multi-stage Dockerfile with non-root user, docker-compose with 4 services and health checks, Makefile convenience targets.

🔮 Possible next phases: API key authentication, GitHub Actions CI/CD, HTTPS reverse proxy (Caddy), OpenAI production switch, Grafana dashboard.
